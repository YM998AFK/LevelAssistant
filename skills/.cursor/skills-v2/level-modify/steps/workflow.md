# 改关主 Agent · 执行流程

---

## §1 路径判断 + 轻量修复

### 标准路径 vs 轻量修复

| 情况 | 判断条件 | 走哪条路径 |
|------|---------|----------|
| **标准路径** | 新 session，或当前 session 尚未分析过该包 | 完整 4 步（Step 1→2→3→4） |
| **同 session 轻量修复** | 当前 session 刚完成该包的生成/修改，且设计师指出具体问题 | 免除 S1/S2/S3 扫包，直接出轻量确认单 |

> 判断不了 → 默认走**标准路径**。

### 轻量修复确认单格式

```
【问题】<设计师指出的问题，一句话>
【改法】<agent 打算怎么改，一句话>
【改后效果】<改完后这段演出是什么样>
确认？(y/n)
```

**轻量修复中不可免除**：轻量确认单（M7不免）、派改造员、派审查员。

### 快速推断规则（以下情况直接进入 Step 2，无需追问）

| 用户提供 | 推断结论 | 对应模式 |
|---------|---------|---------|
| zip + 答案页.cpp（含 `▼▲` 折叠标记，或文件名含"答案页"） | 修正关卡代码/折叠区与 cpp 对齐 | M-10 / M-11 |
| zip + "改 N 范围" / "范围改成 a~b" | 调整循环上界 | M-4 |
| zip + "换[角色名]" | 族内替换主角 | M-6 |
| zip + "练习X改成练习Y" | cid 切分 | M-1 |
| zip + "再加一关" / "加挑战N" | 同族加关 | M-7 |

附加规则：
- 提供 cpp 文件数 ≥ 2，且文件名分别对应不同挑战 → 默认所有对应挑战都要对齐，不询问
- 用户说"按你的推测" → 直接给出推断并说明理由，等用户否定后再调整
- 如需追问，**一次性列出所有疑问**，不轮番追问

## §2 Step 2 · 扫包 + 改关决策

### 扫包

读取并执行扫包模块（与读 `modify_playbook.md` 同批次并行）：

```
读 `../../_shared/scan_package.md`
变量：
  EXTRACTED_DIR_PATH = <extract_archive 返回的 dest 绝对路径>
  WS_FILE_PATH       = <EXTRACTED_DIR_PATH 内的 .ws 绝对路径>
  SKILL_DIR          = <系统提示中的 Skill 目录绝对路径>
```

**有包体必扫**：有 zip/.ws 输入且当前 session 未扫过该包时必须执行。唯一豁免：完整扫描结果已在当前 session 上下文中。

### 读 modify_playbook.md（与扫包并行，对比卡前必须完成）

1. 读 `../../_knowledge/modify_playbook.md` 全文
2. 对照 §1 模式总览，匹配本次任务对应的模式代号（M-1~M-11）
3. 读该模式的"最小 diff"小节，将约束纳入改动清单

未完成此步 → 禁止输出对比卡。

### 需要资源查找时

若改造方案涉及替换角色/换物件/查动画/查场景，强制在扫描阶段同步派遣资源搜索员：
- 读 `agents/resource_agent.md`，填写任务后派遣
- 与扫描员**同一批次**派遣

---

### 状态对比卡格式（M7 卡点后发出）

> 对比卡基于【语义转换】的结论填写，聚焦于**变化点**，不重复描述不变的内容。正文不得出现积木名、路径、fragment 编号等技术细节。
>
> ⚠️ **扫包、语义转换、模式匹配等中间分析过程禁止输出给用户**——直接输出对比卡，其余内容保留在内部。

```
【当前】<一句话：玩家输入什么，看到什么结果>
【改后】<一句话：同样输入，改后看到什么结果（变化部分加粗）>
【演示值】当前 <值> → 改后 <值>

确认？
```

> M7 强制卡点：对比卡发出后，必须等人类明确回复"确认/可以/OK"才允许进入 Step 3。

## §3 Step 3 · 派改造员

### 如何派遣改造员子 Agent

> ⚠️ **禁止把 `modifier_agent.md` 全文复制进 task**——改造员会自行读取规范文件。主 Agent 只发简短任务卡（50 行以内）。

直接调用 `run_subagent`，task 填写如下模板（将 `<...>` 替换为实际值）：

```
你是改造员子 Agent。

请先读取你的完整规范文件（所有约束、步骤、工具白名单都在里面）：
`<Skill目录>/level-modify/agents/modifier_agent.md`

本次任务变量：
- WS_FILE_PATH    = <工作目录下 uuid.ws 的绝对路径>
- WORKDIR_PATH    = <解压工作目录绝对路径>
- BASELINE_ZIP    = <母本 zip 绝对路径>
- OUTPUT_ZIP_PATH = <产出目录>/modify/<原包名>-v<N>.zip  （N 递增，不覆盖旧版）

改动清单：
改动1  路径：<scene.children[...].fragments[...]...>
       改前：<具体积木类型和参数值>
       改后：<目标值>
       工具：<MCP 工具名>
改动2  ...
```

**改动路径规则**：路径要精确到具体 block（不能写"控制台的某处"）；改前值来自扫描员 §5/§6/§7 的输出；参照 `../../_knowledge/modify_playbook.md` M-1~M-10 的"最小 diff"。

### 改造员返回 paused 时的处理

改造员返回 `{"paused": true, "resume_id": "...", "question": "..."}` 时：

1. 读取 question（失败工具名、报错、已完成步骤）
2. 主 Agent 决策：
   - 有明确解法 → 调用 `resume_subagent(resume_id, instruction)` 注入指示
   - 需要人类判断 → 向设计师说明后再 `resume_subagent`
3. resume 后改造员从断点继续；若再次 paused，会返回新 resume_id，继续按上述流程

> `resume_subagent` 只能调用一次，调用后原 resume_id 失效。

### 改造员返回后的质检五问

收到改造员返回（非 paused），以下五项必须全 YES 才进入 Step 4：

1. 改造前确认表存在且每个改动都有列出？
2. 每个改动都有"[改动N] 完成"的执行回报？
3. validate_workspace error_count == 0？
4. verify_gates.py 输出中 gate1/gate2/gate3 均 PASS（或 FAIL 与母本一致）？
5. verify_gates.py 输出中 gate4（zip 完整性）和 gate5（积木节点类型归属）均 PASS 或与母本 FAIL 一致（遗留）？

任一 NO → 主 Agent 决策修法，重新派改造员（新开 Task）。

## §4 Step 4 · 派审查员 + 交付

### 如何派遣审查员子 Agent

> ⚠️ **禁止把 `reviewer_agent.md` 全文复制进 task**——审查员会自行读取规范文件。主 Agent 只发简短任务卡（60 行以内）。

直接调用 `run_subagent`，task 填写如下模板（将 `<...>` 替换为实际值）：

```
你是独立审查员子 Agent。

请先读取你的完整规范文件（所有检查清单、约束都在里面）：
`<Skill目录>/_shared/reviewer_agent.md`

本次任务变量：
- MODE          = 修改
- SKILLS_DIR    = <Skill 目录绝对路径>
- NEW_ZIP       = <Step 3 输出的新包绝对路径>
- BASELINE_ZIP  = <母本 zip 绝对路径>

step 变更表（改后效果摘要）：
| step | 状态 | 说明 |
|------|------|------|
| <改动项> | 变更 | <简要说明> |
| 其余 handler | 同 | 未改动 |

改动路径断言表（来自改造员执行回报，原样复制）：
| # | 精确 ws 路径 | 改前值 | 改后值 |
|---|------------|--------|--------|
| 1 | <路径> | <改前> | <改后> |

答案页 cpp 原文（用于语义对照，无则填"无"）：
<直接粘贴 cpp 文件内容>
```


### 审查员返回后的质检三问

1. `总体：PASS` 字面存在？
2. 第6项（MUST/NEVER）和第7项（观赏性）所有子条目都是 PASS？
3. M3/M6/N2/N6/N7/N8/N9/N12 的具体结论已在返回报告中？

全 YES → 进入交付。任一 NO → 重新派改造员（新开）→ 重新派审查员（新开），循环直到全 PASS。

### 路径与版本号自查

- 改造员的 `${OUTPUT_ZIP_PATH}` 填写全局产出目录下的绝对路径
- 不覆盖旧版：`-v{N}.zip` 已存在则递增为 `v{N+1}`
- 改造员打包后无需再调用 `create_archive`（否则产生 zip 嵌套）

### 交付消息格式

```
## 交付

包文件   : output/modify/{原包名}-v{N}.zip (<大小> KB)
改动摘要 : （引用 step 变更表，不超过 5 条）
遗留问题 : <有则列 / 无则写"无">
verify_gates : PASS(5/5) gate1✅ gate2✅ gate3✅ gate4✅ gate5✅
审查员结论 : 总体 PASS（M3: <具体> / N7: <具体> / 观赏性: <具体>）

MUST/NEVER 自检：M1-8 ✅ / N1-12 ✅
（🤖 verify_gates PASS / 🤖 MCP validate PASS / 🔍 审查员第6~7项 PASS / 👀 人工已确认 M4/M7）
```

审查员报告里 M3/M6/N2/N6/N7/N8/N9 以及观赏性的具体结论，必须在交付消息里转述，不能只写"审查员 PASS"。

**输出精简上限**：交付摘要 ≤15 行；确认单 ≤10 行；每条追问说明 ≤3 行。
