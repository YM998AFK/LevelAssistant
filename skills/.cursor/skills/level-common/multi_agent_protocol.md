# 多 Agent 协作规范（批量改造 / 长任务必读）

> **触发条件**：单次任务涉及 **≥2 个关卡** 改造，或用户明确要求并行分发时读本文件。
> 单关卡改造直接看本文末"单关卡拆法"即可，不需要读全文。

过去踩过的坑：把"扫母本"和"写代码"揉在一起，上下文越长越糊涂；或者主 agent 自己总结用户需求喂给子 agent，信息一路失真；或者审查员给"建议"而不是给事实，主 agent 照抄建议反而把代码改坏。本节规定多 Agent 协作的强制契约。

## 核心理念：主 agent 是项目经理，不是码农

主 agent 的 4 项不可委派职责：

1. 跟用户对话、澄清歧义（子 agent 没有与用户对话的权限）
2. 任务拆解 + 亲自写子 agent 的 prompt（保证**用户原话透传**）
3. 验收（对照契约逐字段检查子 agent 返回）
4. 给用户汇总交付消息

**例外**：单关卡改造时，主 agent 可以亲自下场改代码（因为没有并行任务要管理，上下文不会被路由占满）。见下方"单关卡拆法"。

## 三种子 agent 及职责契约

| 类型 | `subagent_type` | `readonly` | 输入 | 输出 | 禁止 |
|---|---|---|---|---|---|
| **扫描员** | `explore` | true | 母本路径 + 扫描要点列表 | cid / 变量 / 改动点 事实清单（**必须按 level-modify 0.1.a 分 A 栏活代码 / B1 候补模块 / B2 真遗留 三栏，不得混写**；**关卡运行模式识别（云编译 / OJ，见 level-modify 0.1.c）写在扫描报告第一行**）+ **运行时变量覆盖影响表**（见 level-modify 0.1.b） | 写文件、做决策、推测需求、**把孤立 fragment 当活代码汇报**、**忽略 SetVar / ListReplaceItemAt 对下游循环/条件的覆盖**、**把 `WhenReceiveMessage("参数注入")` 当作普通事件处理器归到 A 栏**（它是受保护 hat，必须归 B1） |
| **改造员** | `generalPurpose` | false | 用户原话 + 扫描报告 + 硬约束 | 结构化交付报告 + verify PASS + 嵌套审查 PASS | 越界改其他关卡、跳过 verify、自己总结需求 |
| **审查员** | `explore` | true | 母本 + 新包 + 审查项 | PASS/FAIL 清单 + 证据 | 写"建议"、给中间态、改文件 |

**类型不许混用**：改造不能用 `explore`（要写文件），审查不能用 `generalPurpose`（同一 type 容易继承主 agent 盲区）。

## 单关卡改造的最小拆法（2 个子 agent）

```
主 agent（项目经理 + 码农双角色）
  ├─ 阶段 1：跟用户对齐改动点（主 agent 自己做）
  ├─ 阶段 2：派 1 个 explore 扫描员 ─► 返回母本事实清单
  ├─ 阶段 3：主 agent 自己改代码 + 跑 verify_blocks.py
  └─ 阶段 4：派 1 个 explore 审查员 ─► PASS/FAIL
            FAIL → 修 → 再新开一个审查员（不 resume），循环到 PASS
```

两步绝对不能省：

- **扫描必须派子 agent**。例外：母本 < 100 行且无分支变量时，主 agent 可自扫。
- **审查必然不许主 agent 自审**。刚改完自己看自己必然放水。

## 多关卡批量改造的拆法（N 个关卡 → 3N 个子 agent）

```
用户："把 A/B/C 三个关卡都改成 XX 模式"
  │
  ▼
【阶段 1 理解+澄清】主 agent 自己做
  - 问："A/B/C 改法一样吗？"
  - 问："以哪个母本为基准？"
  - 对齐到"每个关卡的改动点清单"
  ▼
【阶段 2 前置扫描】并行派 3 个 explore 扫描员
  ├─ 扫 A 母本 ─► 事实清单 A
  ├─ 扫 B 母本 ─► 事实清单 B
  └─ 扫 C 母本 ─► 事实清单 C
  主 agent 合成一张"改动计划表"
  ▼
【阶段 3 改造派发】并行派 3 个 generalPurpose 改造员
  ├─ 改 A（输入 = 用户原话 + A 事实清单 + 硬约束）
  │    └─ 改造员自己嵌套 1 个 explore 审查员
  ├─ 改 B ...
  └─ 改 C ...
  ▼
【阶段 4 验收+汇总】主 agent 自己做
  - 任一 FAIL → 打回改造员重做（主 agent 不亲自修）
  - 全 PASS → 给用户一张 N 行交付表（每行一个关卡 + 证据）
```

**冲突预防**：N ≥ 3 的并行改造，优先使用 `best-of-n-runner` 子 agent，每个关卡一个独立 git worktree，避免对同一文件写冲突。

## 派子 agent 的 prompt 契约（5 字段，缺一不准派）

主 agent 在派改造员/审查员时，prompt 必须严格包含以下 5 节，**顺序固定，标题不变**：

```markdown
## 1. 用户原话（原文复制，禁止总结）
<整段粘贴用户说的话，zh-CN 原文，一个字都别改>

## 2. 本次负责的对象
- 母本路径：input/...
- 目标输出：output/modify/<name>-v{N}.zip

## 3. 必读的 SKILL 章节编号
- 修改已有关卡工作流 → 0.1 ~ 0.6
- 最小化改动原则
- 路径规范

## 4. 交付物硬约束（字面量，不允许自由发挥）
- 输出路径：output/modify/<name>-v{N}.zip
- 必须跑 python scripts/verify_blocks.py <zip> 且结果 PASS
- 必须派 explore 审查员（0.6 ④），报告落 reports/<name>-review.md
- 交付前必须看到审查员"总体判定：PASS"

## 5. 禁止行为
- 不许改母本的 cid / mode 分支结构（除非用户原话里明确授权）
- 不许自己总结或"优化"需求；有疑问列成"未决问题"返回主 agent
- 不许越界改其他关卡（只负责本次分配的那个）
- 不许用主 agent 自己的校验脚本替代审查员
```

**规则**：子 agent 若发现主 agent 漏了任一字段，应立刻返回"prompt 不合契约，第 N 字段缺失"，不开工。

## 子 agent 返回的结构化模板

**改造员返回：**

```markdown
## 改造结果
- 输出路径: <绝对路径>
- 改动文件: <.ws 文件名>
- 改动点列表:
  1. <具体描述>
  2. ...

## verify 结果
- 命令: python scripts/verify_blocks.py <zip>
- 输出（最后 10 行原样粘贴）：
  <...>
- 判定: PASS / FAIL

## 审查报告
- 路径: reports/<name>-review.md
- 总体判定: PASS / FAIL

## 未决问题
- [ ] <问题 1>（没有就写"无"，禁止留空）
```

**审查员返回**：严格按 0.6 ④ 模板（每行 `项目 N <名> [PASS|FAIL] 证据：...`，结尾 `总体判定：PASS/FAIL`）。

**硬规则**：子 agent 若返回散文或字段缺失 → 主 agent **打回重做，不许自己补**。

## 5 道信息防丢防线

| # | 防线 | 落地做法 |
|---|---|---|
| 1 | 用户原话透传 | prompt 第 1 字段**整段粘贴**用户原话，禁止主 agent 加工改写 |
| 2 | 结构化返回 | 子 agent 散文返回 → 主 agent 打回重做，不猜 |
| 3 | 新开不 resume | 每次复审都新开 Task 调用，**不传 resume 参数**（防止继承上次放水判断） |
| 4 | 换 type 换盲区 | 改造用 `generalPurpose`，审查用 `explore + readonly=true`，风格互不同化 |
| 5 | 主 agent 只看契约 | 主 agent **不读子 agent 中间思考**，只读结构化字段里的 PASS/FAIL + 证据 |

## 主 agent 的"质检员"强制动作

收到每个子 agent 返回后，**必须依次核对以下 3 项**，缺一不准继续/交付：

1. ✅ 返回里 `verify 判定: PASS` 字面存在？否 → 打回
2. ✅ 审查报告文件真的存在 + `总体判定：PASS` 字面存在？否 → 打回
3. ✅ 给用户的交付消息是**逐关卡列证据的汇总表**，不是"都做完了"这种散文

**任何一步跳过 = 主 agent 失职**，等同于交付未验收的代码。

## 容易踩的坑（对照表）

| 坑 | 防范 |
|---|---|
| 把"扫 A + 改 A + 审 A"塞给**同一个**子 agent | 必须拆三段，扫描/改造/审查用**不同** agent |
| 主 agent 把"我以为的需求"喂给子 agent | prompt 第 1 字段必须粘贴**用户原话**，不总结不改写 |
| 收到 FAIL 但觉得"差不多就算了"直接交付 | 0.6 ④ 硬规则：FAIL 必须修 + 新开审查员复审 |
| 复审时 `resume` 旧审查员省 token | Task 调用**不传 resume 参数**，永远新开 |
| 并行改造对同一文件写冲突 | N ≥ 3 用 `best-of-n-runner`，每个子 agent 独立 git worktree |
| 审查员给"建议修改 XX"，主 agent 照抄 | 审查员 prompt 里硬禁止建议；主 agent 只读 PASS/FAIL + 证据 |
| 主 agent 对话很长后，亲自下场改第 2 个关卡 | 多关卡场景主 agent 严格只做路由+质检，一个代码不写 |
